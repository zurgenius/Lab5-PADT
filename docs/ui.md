# Графический интерфейс

## Назначение

Dear ImGui demo показывает все этапы лабораторной в одном приложении:

```text
C++ DSL -> AST -> QueryOptimizer -> CSV/SQLite backend -> QueryResult
```

Интерфейс предназначен для демонстрации структуры запроса, работы правил оптимизации, перевода запроса в backend и результата на реальных данных. Это не редактор произвольного DSL: запросы создаются в C++ и представлены готовыми сценариями.

## Зависимости и сборка

Общие требования:

- компилятор с поддержкой C++20;
- CMake 3.20 или новее;
- Make для команд из `Makefile`;
- заголовки и библиотека SQLite3, доступные для `find_package(SQLite3)`;
- OpenGL и системные библиотеки оконной подсистемы;
- утилита `sqlite3` для `make demo-data` и `make program`;
- доступ к интернету при первой GUI-сборке.

GoogleTest, GLFW и Dear ImGui загружаются через CMake `FetchContent`. Noto Sans загружается из зафиксированной ревизии Google Fonts, проверяется по SHA-256 и кешируется в `build/assets`. Повторная конфигурация использует локальный файл, если его хеш совпадает.

Для Arch Linux обычно нужны пакеты группы `base-devel`, `cmake`, `sqlite`, `mesa` и библиотеки X11, необходимые GLFW: `libx11`, `libxrandr`, `libxinerama`, `libxcursor`, `libxi`. Для других дистрибутивов названия пакетов отличаются.

В Windows рекомендуется собирать через CMake и Visual Studio Build Tools. Текущий `Makefile` использует POSIX-команды `rm` и `cp`, поэтому без MSYS2 или аналогичного окружения следует использовать ручные CMake-команды. SQLite3 должен быть установлен так, чтобы CMake мог найти его заголовки и библиотеку.

Основной запуск:

```bash
make run
```

Команда пересоздает `data/demo.sqlite3`, конфигурирует GUI, собирает `program` и запускает его.

Ручная сборка:

```bash
cmake -S . -B build -DWITH_RELALG_GUI=ON
cmake --build build --target relalg_program
./build/program
```

При ручной сборке существующая `data/demo.sqlite3` не пересоздается. Для обновления базы следует отдельно выполнить:

```bash
make demo-data
```

Консольная сборка без ImGui:

```bash
cmake -S . -B build-console -DWITH_RELALG_GUI=OFF
cmake --build build-console --target relalg_program
```

## Начальное состояние

После запуска открывается вкладка `Backends`, потому что она показывает полный сценарий лабораторной. По умолчанию:

- выбран CSV backend;
- оптимизатор включен;
- открыто представление `Исполнение`;
- демонстрационный запрос уже выполнен;
- блок исходных документов свернут.

Главная навигация содержит три вкладки: `DSL/AST`, `Optimizer` и `Backends`.

## Вкладка DSL/AST

Вкладка показывает построение и разные представления исходного AST без исполнения запроса.

Слева находится список готовых примеров:

- `users projection` - структурированный `Select` и `Project`;
- `users join orders` - `Join` и итоговый `Project`;
- `renamed students` - `Rename`, raw-условие и `Project`;
- `push-down selection` - пример для проталкивания фильтра под `Join`;
- `projection pruning` - пример удаления ненужных колонок перед `Join`.

Справа доступны представления выбранного примера:

- `DSL` - исходная запись запроса на C++ DSL;
- `RA` - компактная нотация реляционной алгебры;
- `AST` - дерево операций с дочерними узлами;
- `Schema` - схема результата выражения.

Выбранный пример сохраняется при переходе во вкладку `Optimizer`. Изменение примера не влияет на вкладку `Backends`, у которой есть отдельный фиксированный запрос.

## Вкладка Optimizer

Оптимизатор автоматически применяется к примеру, выбранному в `DSL/AST`. Дополнительная кнопка запуска не требуется.

Доступны три представления:

- `RA до/после` сравнивает нотацию исходного и оптимизированного запросов;
- `AST до/после` сравнивает структуру деревьев;
- `Отчет` показывает примененные правила и предупреждения.

Для демонстрации push-down следует выбрать `push-down selection`. В оптимизированном дереве `Select(users.age > 20)` перемещается в левую ветку `Join`.

Для демонстрации projection pruning следует выбрать `projection pruning`. Оптимизатор добавляет проекции к сторонам `Join`, сохраняя выбранные колонки и оба join-ключа.

Пример `renamed students` использует raw-условие `score >= 80`. Оптимизатор не разбирает raw-текст и показывает соответствующее предупреждение в отчете.

## Вкладка Backends

Эта вкладка использует фиксированный end-to-end запрос:

```cpp
Table("users", ...)
| Join(Table("orders", ...), "users.id", "orders.user_id")
| Sigma(Gt("users.age", Literal(20)))
| Pi({"users.name", "orders.total"})
```

Запрос соединяет пользователей с заказами, оставляет пользователей старше 20 лет и выводит имя и сумму заказа.

### Элементы управления

`CSV` и `SQLite` выбирают backend исполнения. При переключении представление результата обновляется для выбранного backend.

`Использовать оптимизатор` определяет, какой AST передается backend:

- включено - исполняется результат `QueryOptimizer`;
- выключено - исполняется исходное дерево.

Оба варианта должны возвращать одинаковые строки, но CSV execution plan и сгенерированный SQL могут иметь другую структуру.

`Запустить` активирует выполнение текущей конфигурации. При старте demo выполнение уже активно; переключение backend или оптимизатора сразу отображается в текущем представлении.

### Представления потока

`DSL и AST` показывает текст фиксированного DSL-запроса и исходное дерево.

`Оптимизация` показывает RA-нотацию до и после оптимизации, а также список примененных правил.

`Исполнение` зависит от выбранного backend.

Для CSV отображаются:

- последовательность чтения CSV-файлов;
- фильтрация;
- проекции входов;
- nested-loop join;
- итоговая проекция;
- таблица `QueryResult`.

Для SQLite отображаются:

- сгенерированный параметризованный SQL;
- значения bind-параметров;
- таблица `QueryResult`.

Колонки результата можно изменять по ширине. Область строк имеет вертикальную прокрутку.

Блок `Исходные документы и таблицы` раскрывает данные `users` и `orders`, прочитанные из CSV. Он позволяет сопоставить входные строки с результатом обоих backend. SQLite содержит эквивалентный набор данных из `data/demo_seed.sql`.

## Демонстрационные сценарии

### Сценарий 1: DSL и AST

1. Открыть `DSL/AST`.
2. Выбрать `users projection`.
3. Сопоставить запись в `DSL` с деревом в `AST`.
4. Проверить порядок операций в `RA`.
5. Проверить итоговые колонки в `Schema`.

### Сценарий 2: оптимизация

1. В `DSL/AST` выбрать `push-down selection`.
2. Открыть `Optimizer`.
3. Сравнить `RA до/после` и `AST до/после`.
4. В `Отчет` проверить правило проталкивания `Select`.
5. Повторить сценарий с `projection pruning`.

### Сценарий 3: CSV и SQLite

1. Открыть `Backends` и `Исполнение`.
2. Выбрать `CSV` и оставить оптимизатор включенным.
3. Просмотреть план и четыре строки результата.
4. Раскрыть исходные документы и сопоставить пользователей с заказами.
5. Выбрать `SQLite`.
6. Проверить SQL, bind-параметр возраста и эквивалентный результат.
7. Отключить оптимизатор и сравнить форму backend-запроса, не изменяя результат.

## Ошибки и ограничения UI

Ошибки исполнения CSV и SQLite перехватываются и показываются красным текстом во вкладке `Исполнение`. Типичные причины:

- отсутствует `data/users.csv`, `data/orders.csv` или `data/demo.sqlite3`;
- схема CSV не содержит колонку из AST;
- CSV-строка имеет неверное число полей или незакрытую кавычку;
- SQLite не может открыть базу или подготовить запрос;
- backend получает raw-условие, которое нельзя безопасно разобрать.

Ошибка инициализации GLFW выводится в `stderr`. Если окно создать невозможно, программа завершается с кодом `1`.

Если скачанный шрифт отсутствует во время запуска, ImGui использует встроенный шрифт. Приложение продолжает работать, но кириллица может отображаться некорректно. При нормальной сборке CMake проверяет наличие и хеш Noto Sans.

Текущие ограничения:

- UI не редактирует и не компилирует введенный пользователем текст DSL;
- список примеров задается в C++;
- вкладка `Backends` исполняет только один фиксированный запрос;
- backend поддерживают операции текущего AST, без aggregate, group, sort и limit;
- raw-условия можно показать в DSL/AST, но нельзя исполнить;
- настройки ImGui могут сохраняться локально в `imgui.ini`.

## Расширение UI

UI сейчас находится в `src/main.cpp` и разделен на функции рендеринга.

Чтобы добавить DSL/AST-пример:

1. создать функцию `build...()`, возвращающую `ExprPtr`;
2. увеличить `exampleCount()`;
3. добавить имя в `exampleName()`;
4. добавить DSL-текст и выражение в `makeExample()`.

Такой пример автоматически станет доступен во вкладках `DSL/AST` и `Optimizer`.

Чтобы расширить backend-demo:

1. зарегистрировать источник в `makeDemoCatalog()`;
2. добавить данные в `data/` и `demo_seed.sql`;
3. построить исполняемый AST в `renderBackendsTab()`;
4. вернуть результат через общий `QueryResult`;
5. использовать `renderResultTable()` для одинакового отображения разных backend.

Для новой основной стадии следует добавить отдельную функцию `render...Tab()` и новый `BeginTabItem` в главной панели. Если UI продолжит расти, функции и состояние следует вынести из `main.cpp` в отдельный UI-модуль, оставив в `main` только инициализацию GLFW/ImGui и главный цикл.

Обработка нового backend должна сохранять общий контракт: входной `ExprPtr`, выходной `QueryResult`, диагностическая информация для представления `Исполнение` и исключение с понятным сообщением при ошибке.
