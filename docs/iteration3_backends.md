# Итерация 3: CSV и SQLite backends

## Цель

Третья итерация замыкает демонстрационный поток проекта:

```text
C++ DSL -> AST -> QueryOptimizer -> CSV/SQLite backend -> QueryResult
```

Один AST исполняется над CSV-файлами или компилируется в параметризованный SQL для SQLite. Оба backend возвращают общий `QueryResult`, поэтому результат можно одинаково показать в UI и сравнить в тестах.

## Общая модель

Публичные типы расположены в `include/backend`:

- `Cell` хранит строковое значение или `NULL`;
- `Row` хранит `Sequence<Cell>`;
- `QueryResult` хранит схему и `Sequence<Row>`;
- `SourceCatalog` связывает имя из `Table(...)` с CSV-файлом или таблицей SQLite;
- `PredicateEvaluator` вычисляет структурированные условия DSL.

В публичном API используются `Sequence<T>`, внутреннее владение построено на `MutableArraySequence<T>`. `std::vector` не используется. Нетемплейтная реализация находится в `src/backend/*.cpp`.

## CSV backend

`CsvBackend` рекурсивно исполняет узлы AST:

- `Table` читает строки через существующий `FileLineReadOnlyStream`;
- `Select` применяет `PredicateEvaluator`;
- `Project` перестраивает строки по схеме проекции;
- `Join` выполняет inner nested-loop join;
- `Rename` сохраняет данные дочернего выражения.

CSV parser поддерживает заголовок, запятые внутри quoted-полей и экранирование `""`. Многострочные поля первой версии не поддерживаются.

Результат исполнения дополнен планом операций: чтение файлов, фильтрация, join и projection.

## SQL и SQLite

`SqlCompiler` превращает каждый AST-узел во вложенный `SELECT`. Имена таблиц и колонок экранируются как SQLite identifiers. Литералы не встраиваются в текст SQL: компилятор создает `?` и отдельный список `SqlParameter`.

`SQLiteBackend`:

1. компилирует AST через `SqlCompiler`;
2. открывает базу в read-only режиме;
3. создает prepared statement;
4. привязывает типизированные параметры;
5. материализует строки в общий `QueryResult`.

## Демонстрационные данные

В `data/` находятся:

- `users.csv` и `orders.csv`;
- `demo_seed.sql` со схемой и данными SQLite;
- `demo.sqlite3`, готовая база для запуска UI.

База пересоздается командой:

```bash
make demo-data
```

CSV и SQLite содержат одинаковые сущности, поэтому end-to-end тест исполняет один оптимизированный запрос обоими backend и сравнивает результаты. Вещественные значения сравниваются численно: SQLite может вернуть `120.5`, когда CSV хранит `120.50`.

## UI

Вкладка `Backends` показывает полный поток на запросе с `Join`, `Select` и `Project`:

- исходный DSL и AST;
- RA до и после оптимизации и отчет правил;
- выбор CSV или SQLite;
- переключатель исполнения исходного или оптимизированного AST;
- CSV execution plan либо сгенерированный SQL и bind-параметры;
- итоговую таблицу;
- исходные документы `users` и `orders`.

## Ограничения

- поддерживается inner equi-join;
- нет aggregate, group, sort и limit;
- raw-условия не разбираются и не исполняются;
- один SQL-запрос работает с одной SQLite-базой;
- CSV backend материализует промежуточные результаты и правую сторону join;
- SQL dialect ориентирован на SQLite.
